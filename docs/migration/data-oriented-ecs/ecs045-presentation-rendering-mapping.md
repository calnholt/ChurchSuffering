# ECS-045 presentation and rendering migration mapping

This is the owned reconciliation for ECS-045. The implementation replaces display-system-owned draw state with deterministic presentation updates, reusable render-packet layers, typed external effect/audio requests, and read-only draw consumers. Catalog initialization, diagnostics formatting, and capacity growth may allocate; warmed update, extraction, sorting-stable, queue, and draw paths do not.

## Runtime responsibility map

- Transform, tween, hierarchy, and parallax: `PositionTweenPresentationSystem`, `ParallaxPresentationSystem`, and `TransformResolver`.
- Rendering: `RenderPacketStore` owns reusable per-layer buffers. Membership or Z/stable-order changes mark a layer dirty; animation-only packet writes do not re-sort it.
- Card, hand, actors, HUD, tooltip, modal, overlay, effect, and shader display responsibilities collapse into packet kinds and layer selection in extraction.
- Visual effects and shader/audio boundaries use unmanaged request components/events and established-capacity `PresentationRequestQueues`.
- `PresentationEventHub.BuildRoutes` exposes all 38 typed routes for composition into the single root `EventRoutingEndpoint`; its `Attach` method remains an isolated-test convenience.
- Diagnostics use reusable profiler/entity-inspector arrays; the isolated `NewWorldSnapshotFixtureHost` registers all 46 fixture IDs from the current `DisplaySnapshotRegistry` without capture, verification, acceptance, or baseline mutation. The canonical fixture and verification reference remains [`docs/display-snapshots.md`](../../display-snapshots.md).
- Draw consumers receive only `ReadOnlySpan<RenderPacket>` data and an external sink. They cannot access or mutate the world, events, commands, packet caches, timelines, or input.
- ECS-040 interaction/modal state, ECS-041 card/hand state, and ECS-042 combat presentation signals are read or consumed at the frozen typed boundaries; gameplay ownership is not duplicated here.
